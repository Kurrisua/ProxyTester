# ProxyTester 下一阶段安全研究平台规划

> 本文档用于规划 ProxyTester 后续从“代理可用性检测平台”升级为“代理安全风险检测、行为分析、动态观测与可视化研究平台”的实现路线。  
> 当前阶段只做规划，不修改业务代码。  
> 最后更新：2026-04-17

## 1. 总体定位变化

现有系统已经完成了免费代理的广泛收集、去重整合、基础可用性检测、协议识别、性能测评、评分、入库、API 查询和前端展示。它当前回答的主要问题是：

1. 代理能不能连通。
2. 代理支持什么协议。
3. 代理速度如何。
4. 代理地理位置和匿名性如何。
5. 代理是否适合作为可用资源被筛选出来。

下一阶段的核心问题要变化为：

1. 这个代理是否可信。
2. 这个代理是否修改、注入或替换了传输内容。
3. 这个代理是否在 HTTPS 或 SOCKS 场景下存在中间人攻击风险。
4. 这个代理是否只在特定条件或多轮访问后触发恶意行为。
5. 这个代理的行为能否被归类、评分、追踪和可视化分析。

因此，后续系统不应只输出“可用/不可用”或“快/慢”，而应输出“正常/可疑/恶意”的行为分类，以及可解释的证据链。

## 2. 对现有项目的批判性分析

### 2.1 已具备的有利基础

当前项目并不是从零开始，已有几个适合继续扩展的基础：

1. 已有完整的代理采集、去重、检测、评分和入库主流程。
2. `CheckPipeline` 已经支持基础检查器、安全检查器和评分器分层执行。
3. `SecurityResult` 已经具备 `risk_level`、`risk_tags`、`evidence` 等字段，可以承载安全检测结果。
4. `ProxyModel` 已有 `security_risk`、`security_flags`、`security_evidence` 等安全字段。
5. `security/plugins/` 下已经预留了 `honeypot_checker`、`dom_diff_checker`、`mitm_checker` 三个插件入口。
6. 前端和 API 已经能展示代理池状态，后续可以逐步扩展为安全分析界面。

这些说明后续设计应该顺着现有插件式架构生长，而不是推翻重写。

### 2.2 当前主要缺口

现有安全插件目前仍是占位实现，真正缺失的是安全检测的基础设施：

1. 缺少可控蜜罐目标，无法可靠判断代理是否篡改内容。
2. 缺少统一访问执行层，无法标准化地获取直连结果和代理结果。
3. 缺少响应内容、资源、证书、跳转链、网络请求等结构化采集能力。
4. 缺少 HTML 文本差分、DOM 差分、资源哈希比对和风险规则引擎。
5. 缺少 HTTPS 证书采集与直连/代理证书对比能力。
6. 缺少多轮检测任务模型，当前更偏单次批量检查。
7. 缺少检测历史、行为事件、资源异常、证书异常等细粒度持久化表。
8. 前端当前仍是代理池展示视角，不足以支撑安全研究分析。

因此，不能直接把大量逻辑塞进 `MitmChecker` 或 `DomDiffChecker`。正确路径应该是先建立可复用的安全检测基础层，再让现有安全插件调用这些能力。

### 2.3 需要警惕的实现风险

后续内容技术价值很高，但复杂度也会明显上升：

1. **误报风险**：很多页面本身会动态变化，如果没有完全可控的蜜罐页面，差分结果会非常嘈杂。
2. **HTTPS 检测风险**：requests、浏览器、系统证书库、代理 CONNECT 行为之间差异较大，证书采集必须单独设计。
3. **高并发风险**：当前批量检测并发较高，安全检测如果加入浏览器、资源下载、多轮访问，会显著增加耗时和资源占用。
4. **数据量风险**：如果保存响应正文、DOM 快照、网络请求和多轮记录，数据库会迅速膨胀，需要区分摘要、证据和原始样本。
5. **浏览器自动化成本**：Playwright 或类似方案能提高真实性，但不适合一开始对全部代理高并发运行。
6. **法律和伦理边界**：系统应只访问自建蜜罐和明确允许的测试目标，不应对第三方真实站点进行侵入性检测。

规划上必须把这些风险前置处理。

## 3. 目标架构

下一阶段建议新增一个安全检测子系统，和现有代理检测流程解耦但可集成。

建议分层如下：

1. 蜜罐目标层：提供稳定、可控、可校验的测试页面和资源。
2. 访问执行层：统一执行直连访问、代理访问、浏览器访问。
3. 证据采集层：采集响应、资源、证书、跳转链、网络请求、DOM 快照。
4. 差分分析层：执行文本差分、DOM 差分、资源差分、证书差分。
5. 风险规则层：把差异转换为可解释的风险标签。
6. 能力路由层：根据代理协议、稳定性、前序结果决定后续检测路径。
7. 漏斗调度层：按低成本到高成本逐层推进检测。
8. 多轮观测层：控制访问轮次、访问条件、时间间隔和行为模拟。
9. 行为建模层：根据多轮结果输出行为分类和安全评分。
10. 持久化层：保存检测记录、行为事件、资源异常、证书异常和汇总状态。
11. API 与前端层：展示安全状态、行为分类、异常证据和趋势分析。

和现有架构的关系：

1. 现有 `security/plugins/honeypot_checker.py` 应成为蜜罐检测入口。
2. 现有 `security/plugins/dom_diff_checker.py` 应调用 DOM 差分和风险规则能力。
3. 现有 `security/plugins/mitm_checker.py` 应调用证书采集和证书对比能力。
4. 现有 `scoring/security_scorer.py` 应升级为综合安全评分器。
5. 现有 `ProxyModel.security_*` 字段用于保存最终汇总，不应承载全部明细。

### 3.1 按代理能力进行检测路由

安全检测不能对所有代理无差别执行同一套检查。不同检测项有明确前置条件：

| 检测项 | 主要适用代理 | 前置条件 | 不适用或应跳过的情况 |
| --- | --- | --- | --- |
| HTTP 蜜罐页面篡改 | HTTP、SOCKS5 | 能访问 HTTP 蜜罐目标 | 代理无法完成 HTTP 请求 |
| DOM 注入检测 | HTTP、HTTPS、SOCKS5 | 页面访问成功且响应为 HTML | 响应失败、MIME 非 HTML、内容为空 |
| 资源替换检测 | HTTP、HTTPS、SOCKS5 | 能稳定获取目标资源 | 资源请求失败或响应不完整 |
| 下载文件篡改 | HTTP、HTTPS、SOCKS5 | 能下载完整测试文件 | 连接不稳定导致文件不完整 |
| HTTPS 证书 / MITM | HTTPS、SOCKS5 | 能建立 TLS 目标连接 | 纯 HTTP 代理通常不适用 |
| 多轮隐蔽行为检测 | 高可用代理、可疑代理、抽样代理 | 基础检测通过，或已出现可疑信号 | 明显不可用代理不应默认进入 |
| 无头浏览器检测 | 高价值代理、可疑代理、小规模抽样 | 能稳定完成页面加载 | 低稳定代理会造成高成本误报 |

因此每个检测模块都需要先回答三个问题：

1. 这个代理是否具备运行该检测的能力。
2. 当前前置条件是否满足。
3. 如果没有运行，是“不适用”“被策略跳过”，还是“执行出错”。

批判性建议：

1. 不要把“未检测”解释成“安全”。例如没有执行 MITM 检测，不等于没有 MITM 风险。
2. 安全结论需要区分 `normal`、`anomalous`、`not_applicable`、`skipped`、`error`。
3. 数据库和前端都必须显示跳过原因，否则漏斗式检测会让结果产生歧义。

### 3.2 漏斗式逐层检测模式

推荐将后续安全检测设计为漏斗式流程：

```text
第 0 层：采集与去重
  -> 第 1 层：基础连通性检测
    -> 第 2 层：协议与能力识别
      -> 第 3 层：轻量蜜罐与 hash 对比
        -> 第 4 层：DOM / HTML 结构化差分
          -> 第 5 层：资源完整性检测
            -> 第 6 层：HTTPS / SOCKS MITM 检测
              -> 第 7 层：多轮条件触发检测
                -> 第 8 层：无头浏览器深度检测
                  -> 第 9 层：行为分类与安全评分
```

这种模式的核心是：先用低成本、高覆盖的检测处理大量代理，再把高成本检测用于更少、更值得深入分析的代理。

显著优势：

1. 成本更低：浏览器、多轮、资源下载、证书采集都不适合全量高并发运行。
2. 误报更少：基础不稳定的代理先被识别，不会直接进入 DOM diff 或资源替换判断。
3. 解释性更强：每个代理都有清晰路径，能说明它在哪一层通过、异常或被跳过。
4. 更适合调度：低成本检测可以频繁跑，深度检测可以只对可疑代理或抽样代理跑。
5. 更适合前端展示：可以展示代理的检测进度、漏斗层级和失败位置。

潜在代价：

1. 过严的前置过滤可能漏掉低稳定但偶尔作恶的代理。
2. 检测策略会变复杂，需要维护每一层的进入条件。
3. 数据库必须记录 `not_applicable` 和 `skipped`，否则统计会误导。

缓解方式：

1. 对低稳定代理保留小比例抽样进入轻量安全检测。
2. 对来源高风险、历史可疑或异常国家/协议的代理提高检测深度。
3. 对每个检测记录保存前置条件摘要和跳过原因。

## 4. 新增模块规划

### 4.1 蜜罐环境

蜜罐环境是后续安全检测的基准源。它必须优先实现，因为没有稳定基准，就无法可靠判断代理是否作恶。

建议新增模块：

1. `honeypot/`
2. `honeypot/app.py`
3. `honeypot/routes.py`
4. `honeypot/fixtures/`
5. `honeypot/manifest.py`

蜜罐应提供固定页面：

1. 纯静态 HTML 页面。
2. 标准 DOM 结构页面。
3. 表单页面。
4. 多资源页面，包含 CSS、JS、图片。
5. 下载页面。
6. 多外链、多层级复杂页面。

蜜罐应提供固定资源：

1. 图片资源。
2. JavaScript 文件。
3. CSS 文件。
4. 文本文件。
5. ZIP 或伪 ZIP 文件。
6. 伪可执行测试文件。

每个页面和资源都需要有 manifest 描述：

1. 路径。
2. 内容类型。
3. 预期状态码。
4. 预期 MIME 类型。
5. 预期 SHA256。
6. 关键 DOM 选择器。
7. 允许出现的外链域名。
8. 不允许出现的标签或属性。

蜜罐还应记录请求与响应摘要：

1. 请求时间。
2. 请求路径。
3. 请求头。
4. User-Agent。
5. 来源 IP。
6. 响应状态码。
7. 响应头。
8. 响应体哈希。
9. 内容长度。

批判性建议：

1. 第一版不要把蜜罐做成复杂平台，先用 Flask 蓝图或独立 Flask app 实现即可。
2. 蜜罐内容必须尽量静态，避免时间戳、随机数等导致误报。
3. 请求日志可以先落数据库或文件，但响应正文不建议默认全量保存。

### 4.2 双路径访问执行层

后续所有安全检测都依赖“直连结果”和“代理结果”的对比，因此需要统一访问执行模块。

建议新增模块：

1. `security/access/`
2. `security/access/client.py`
3. `security/access/models.py`
4. `security/access/proxy_config.py`
5. `security/access/cert_probe.py`

统一访问接口应支持：

1. 直连访问。
2. HTTP 代理访问。
3. HTTPS 代理访问。
4. SOCKS5 代理访问。
5. 超时控制。
6. 跳转链记录。
7. 响应头采集。
8. 响应体采集或摘要采集。
9. 错误原因结构化返回。

建议的结果对象：

```python
AccessResult:
    success: bool
    mode: "direct" | "proxy" | "browser"
    target_url: str
    final_url: str | None
    status_code: int | None
    response_headers: dict
    body_text: str | None
    body_bytes_sha256: str | None
    body_size: int | None
    mime_type: str | None
    elapsed_ms: float | None
    redirect_chain: list[dict]
    certificate_summary: dict | None
    error_type: str | None
    error_message: str | None
```

批判性建议：

1. 不要让每个 checker 自己写 requests 逻辑，否则证据格式会失控。
2. 第一版可以基于 `requests` 或 `httpx` 完成普通访问，浏览器访问作为后续增强。
3. SOCKS5 支持可能需要额外依赖，必须在依赖和部署文档中明确。
4. 响应体保存要设置大小上限，例如只保存摘要、前 N KB 样本和哈希。

### 4.3 内容差分与 DOM 篡改检测

内容差分层负责把直连和代理访问结果转化为结构化差异。

建议新增模块：

1. `security/diff/text_diff.py`
2. `security/diff/dom_diff.py`
3. `security/diff/resource_diff.py`
4. `security/rules/risk_rules.py`

基础文本差分应检测：

1. HTML 是否整体变化。
2. 是否新增可疑字符串。
3. 是否删除关键内容。
4. 是否新增广告、跳转、追踪相关域名。
5. 是否出现不在 manifest 中的 URL。

DOM 差分应检测：

1. 新增 `<script>` 标签。
2. 新增 `<iframe>` 标签。
3. 新增隐藏元素。
4. 新增广告容器或弹窗容器。
5. 表单 `action` 被改写。
6. 链接 `href` 被改写。
7. 图片、脚本、CSS 等资源地址被替换。
8. 新增 `onclick`、`onchange`、`oninput` 等事件属性。
9. 原有关键节点缺失。

风险规则示例：

| 差异 | 风险标签 | 风险等级 |
| --- | --- | --- |
| 新增外部 script | `script_injection` | high |
| 新增隐藏 iframe | `hidden_iframe` | high |
| 表单 action 被改写 | `form_hijack` | critical |
| 新增广告节点 | `ad_injection` | medium |
| 资源域名被替换 | `resource_replacement` | high |
| 关键正文缺失 | `content_tampering` | medium |

批判性建议：

1. 不要直接比较完整 HTML 字符串作为最终依据，字符串差分只能做线索。
2. DOM 差分必须基于可控页面和 manifest，否则动态页面会造成大量误报。
3. 第一版先覆盖高价值标签：script、iframe、form、link、resource URL、event handler。
4. 风险规则输出必须结构化，便于前端解释和数据库查询。

### 4.4 资源替换与下载篡改检测

资源检测应独立于 HTML 页面检测，因为很多恶意代理只替换下载文件、图片或脚本。

资源检测内容：

1. SHA256 比对。
2. 文件大小比对。
3. MIME 类型比对。
4. 内容摘要比对。
5. JS 文件可疑片段扫描。
6. 下载文件替换检测。

建议风险标签：

1. `file_replaced`
2. `script_modified`
3. `image_rewritten`
4. `css_modified`
5. `suspicious_download_swap`
6. `mime_type_mismatch`

批判性建议：

1. 第一版优先做哈希、大小和 MIME 类型，不要一开始做复杂恶意代码检测。
2. JS 恶意代码识别可以先用规则命中，例如新增外链、eval、document.write、可疑重定向。
3. 下载资源不要使用真实可执行文件，使用伪可执行测试文件即可。

### 4.5 HTTPS / SOCKS MITM 检测

MITM 检测是关键能力，但实现时要拆清楚两件事：

1. 代理通道是否能访问 HTTPS 目标。
2. 代理访问时看到的证书是否与直连基准一致。

证书采集字段：

1. Subject。
2. Issuer。
3. 序列号。
4. 有效期。
5. SHA256 指纹。
6. 公钥算法。
7. 公钥摘要。
8. 证书链摘要。
9. 是否自签名。
10. 校验错误信息。

异常识别：

1. 直连与代理证书指纹不同。
2. 代理返回未知签发者证书。
3. 代理返回自签名证书。
4. 证书链不完整或异常。
5. 证书校验失败但访问层仍接受响应。
6. HTTPS 页面内容和证书同时异常。

风险分类：

1. `tls_normal`
2. `cert_mismatch`
3. `unknown_issuer`
4. `self_signed_cert`
5. `cert_chain_invalid`
6. `mitm_suspected`
7. `tls_validation_bypass`

批判性建议：

1. 不要为了“成功访问”关闭证书校验后直接判定正常。关闭校验只能用于证据采集，不能用于安全结论。
2. 证书采集应使用独立探针，避免被普通 HTTP 客户端封装细节遮蔽。
3. SOCKS5 场景要确认 DNS 解析位置，否则可能影响证书与目标一致性判断。
4. 第一版先检测自建 HTTPS 蜜罐目标，再扩展到少量明确允许的公共测试目标。

### 4.6 多轮检测与隐蔽攻击识别

单轮检测只能识别稳定作恶代理，无法发现随机触发或条件触发攻击。下一阶段必须引入多轮动态观测。

多轮变量：

1. User-Agent。
2. 访问路径。
3. 页面类型。
4. 资源类型。
5. HTTP 与 HTTPS 目标。
6. 请求间隔。
7. 是否使用浏览器访问。
8. 是否触发表单或点击行为。

需要识别的模式：

1. 随机触发型攻击。
2. 条件触发型攻击。
3. 延迟触发型攻击。
4. 协议差异型攻击。
5. 资源定向型攻击。
6. 浏览器专属攻击。

统计输出：

1. 总轮次。
2. 异常轮次。
3. 异常触发率。
4. 首次异常轮次。
5. 异常类型分布。
6. 异常与 UA、路径、资源类型、协议的关联。

批判性建议：

1. 多轮检测不应默认对所有代理运行，否则成本过高。
2. 可以先对高可用代理、可疑代理或抽样代理运行多轮检测。
3. 多轮任务需要和当前同步刷新接口拆开，最好任务化。
4. 多轮结果必须保存明细，否则无法做后续研究分析。

### 4.7 无头浏览器行为模拟

浏览器访问用于发现只针对真实浏览器或 JS 执行后才出现的攻击行为。

建议新增能力：

1. 页面完整加载。
2. 渲染后 DOM 采集。
3. JS 执行后 DOM 采集。
4. 跳转链记录。
5. 网络请求记录。
6. 控制台错误记录。
7. 页面等待。
8. 简单点击。
9. 表单输入。
10. 滚动。

批判性建议：

1. 浏览器检测不能放进第一阶段主路径，成本太高。
2. 第一版应作为“深度检测模式”，只对少量代理触发。
3. 浏览器检测结果要和 requests 级结果区分保存，避免混淆。
4. 需要设置严格超时、页面大小限制和并发限制。

### 4.8 行为分类与风险评分

最终系统需要输出行为分类，而不是只输出单个插件风险。

建议行为分类：

1. `normal`
2. `content_tampering`
3. `ad_injection`
4. `script_injection`
5. `redirect_manipulation`
6. `resource_replacement`
7. `mitm_suspected`
8. `stealthy_malicious`
9. `unstable_but_non_malicious`

建议安全评分输入：

1. 内容篡改风险。
2. 资源替换风险。
3. MITM 风险。
4. 隐蔽攻击触发率。
5. 多轮重复性。
6. 攻击对象敏感度，例如表单、下载、脚本。
7. 证据置信度。

建议输出字段：

1. `security_risk_level`
2. `behavior_class`
3. `security_score`
4. `risk_tags`
5. `evidence_summary`
6. `anomaly_trigger_rate`
7. `last_security_check_at`

批判性建议：

1. 安全评分不能只按标签数量累加，表单劫持和广告注入的严重程度完全不同。
2. 应区分风险等级和置信度。一次轻微 DOM 差异不应直接判定恶意。
3. 最终分类要可解释，必须能追溯到检测轮次和事件证据。

## 8. 实现顺序建议

总体实现顺序应遵循漏斗原则：

1. 先完成低成本、全量可运行的基础能力。
2. 再根据协议能力进入不同安全检测路径。
3. 对可疑代理、高可用代理和抽样代理执行更深层检测。
4. 对每一层都记录执行状态、适用性和跳过原因。

推荐安全检测顺序：

```text
基础连通性检测
  -> 协议识别
  -> 基础性能和稳定性评分
  -> 轻量蜜罐访问检测
  -> 内容 hash 与状态码对比
  -> DOM / HTML 风险规则检测
  -> 资源完整性检测
  -> HTTPS / SOCKS MITM 检测
  -> 多轮条件触发检测
  -> 无头浏览器深度检测
  -> 行为分类与安全评分
  -> 汇总入库与前端展示
```

其中 MITM 检测只对具备 HTTPS 或 SOCKS5 能力的代理执行；多轮检测和浏览器检测不默认全量执行，应对高可用、可疑、高价值或抽样代理执行。

### 第一阶段：安全检测基础设施

目标：建立可靠基准和标准化采集结果。

实现内容：

1. 蜜罐测试环境。
2. 蜜罐 manifest。
3. 双路径访问执行模块。
4. `AccessResult` 结构。
5. 响应摘要、哈希、跳转链采集。
6. 第一版安全检测记录对象。
7. 检测适用性判断模型。
8. 检测执行状态模型。

验收标准：

1. 同一 URL 可以获得直连结果和代理结果。
2. 蜜罐页面和资源有稳定哈希。
3. 检测模块能拿到统一结构的访问结果。
4. 不依赖真实第三方页面即可完成基础对比。
5. 能区分检测成功、异常、不适用、被跳过和执行错误。

### 第二阶段：核心安全检测能力

目标：识别稳定触发的内容篡改、资源替换和 MITM 风险。

实现内容：

1. HTML 文本差分。
2. DOM 差分。
3. 资源哈希、大小、MIME 比对。
4. HTTPS 证书采集。
5. 直连证书与代理证书对比。
6. 风险规则与风险标签。
7. 替换现有 placeholder 安全插件。
8. 按代理协议能力选择检测路径。

验收标准：

1. 新增脚本、iframe、表单 action 修改能被识别。
2. 资源替换能被识别。
3. 证书不一致、自签名、未知签发者能被识别。
4. `SecurityResult` 能输出可解释的 `risk_tags` 和 `evidence`。
5. 纯 HTTP 代理不会被错误标记为“MITM 正常”，而是记录为不适用或未执行。

### 第三阶段：多轮动态观测

目标：识别条件触发和隐蔽型恶意行为。

实现内容：

1. 多轮检测任务。
2. User-Agent 轮换。
3. 页面类型轮换。
4. 资源类型轮换。
5. 请求间隔控制。
6. 异常触发率统计。
7. 异常模式识别。
8. 可疑代理进入深度检测的策略。
9. 低稳定代理的抽样检测策略。

验收标准：

1. 同一代理可以保存多轮检测记录。
2. 能统计异常触发频率和首次异常轮次。
3. 能识别只在特定 UA、路径或资源类型下出现的异常。
4. 能解释代理为什么进入或未进入多轮检测。

### 第四阶段：浏览器行为模拟

目标：检测只在真实浏览器环境或 JS 执行后出现的代理攻击。

实现内容：

1. 引入无头浏览器访问器。
2. 采集渲染后 DOM。
3. 采集网络请求。
4. 采集跳转链。
5. 支持等待、点击、输入、滚动等简单行为。
6. 浏览器检测深度模式。
7. 浏览器检测并发和预算控制。

验收标准：

1. 能区分 requests 级结果和浏览器级结果。
2. 能捕获 JS 执行后的新增节点或跳转。
3. 浏览器检测具备严格超时和并发限制。
4. 浏览器检测只对策略命中的代理执行，不进入全量默认路径。

### 第五阶段：研究数据与可视化

目标：把检测结果沉淀为可分析、可追踪、可展示的研究数据。

实现内容：

1. 数据库表扩展。
2. 安全检测批次记录。
3. 安全检测历史记录。
4. 行为事件表。
5. 行为分类与安全评分。
6. API 扩展。
7. 前端安全总览。
8. 代理安全详情。
9. 异常事件列表。
10. 风险趋势图。

验收标准：

1. 可以按行为类别查看代理。
2. 可以查看某个代理的多轮行为轨迹。
3. 可以查看异常事件及其证据摘要。
4. 可以统计不同国家、协议、来源的异常比例。

## 9. 推荐的最小可用版本

为了避免范围过大，建议第一版 MVP 只做以下内容：

1. 自建蜜罐 HTTP 页面和资源。
2. 直连与代理访问对比。
3. HTML 文本哈希比对。
4. DOM 中 script、iframe、form、event handler、resource URL 差分。
5. 图片、JS、CSS、下载文件的 SHA256 比对。
6. HTTPS 证书指纹对比。
7. 将结果写入现有 `SecurityResult`。
8. 在 `proxies` 表中更新 `security_risk` 汇总。
9. 新增一张安全检测记录表保存摘要。
10. 前端先展示安全风险等级和风险标签。
11. 记录每个检测项的 `applicable`、`not_applicable`、`skipped`、`error` 状态。
12. 对 MITM 等协议相关检测记录明确跳过原因。

MVP 暂不做：

1. 无头浏览器。
2. 复杂多轮调度。
3. 完整证据文件管理。
4. 大型可视化看板。
5. 复杂机器学习分类。
6. 复杂恶意 JS 语义分析。

这样可以先验证核心闭环：代理访问是否改变了蜜罐目标，以及系统能否给出可信证据。

## 实施前置说明：先完成结构稳定化

在正式实现本文档中的蜜罐、双路径访问、DOM 差分、资源替换检测、MITM 检测和多轮观测前，必须先阅读并执行 `docs/project-architecture-audit.md` 中的第 0 阶段建议。

原因是当前项目虽然已经具备代理采集、基础检测、评分、入库、API 和前端展示能力，但后续安全检测需要更细的过程记录和状态语义。如果直接在现有结构上叠加复杂功能，容易出现以下问题：

1. 把“未检测”误当成“安全”。
2. 无法解释某个检测为什么没有执行。
3. 无法保存失败、超时、跳过、不可适用等研究数据。
4. 安全事件无法追溯到具体批次、代理、检测阶段和证据。
5. 前端只能展示结论，无法展示可靠的行为轨迹。

因此，后续实现顺序应当调整为：

```text
结构审计与第 0 阶段稳定化
  -> 完整数据库 schema 与 migration
  -> 检测批次和记录持久化
  -> 蜜罐与双路径访问
  -> 内容差分、资源检测、证书检测
  -> 行为分类、评分、API、前端可视化
```
