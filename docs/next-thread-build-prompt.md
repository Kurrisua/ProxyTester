## 总体目标

当前系统已经具备：

1. 免费代理采集。
2. 代理去重整合。
3. 基础可用性检测。
4. 协议识别。
5. 性能测评。
6. 基础评分。
7. MySQL 入库。
8. Flask API 查询。
9. React 前端展示。

下一阶段目标是把系统升级为：

> 能够检测代理恶意行为、异常行为、隐蔽攻击行为，并能通过数据库、API 和前端进行分析、追踪、解释和可视化展示的代理安全研究平台。

核心能力包括：

1. 蜜罐测试环境。
2. 直连与代理双路径访问。
3. HTML / DOM 差分。
4. 资源完整性检测。
5. HTTPS / SOCKS MITM 检测。
6. 漏斗式逐层检测。
7. 多轮动态观测。
8. 行为分类和安全评分。
9. 安全检测记录入库。
10. 前端安全总览、代理详情、事件列表、检测批次、图表和世界地图。

## 必须先处理的第 0 阶段

在开始实现复杂安全检测前，必须先完成或至少明确处理以下结构问题：

1. 固化状态枚举：检测适用性、执行状态、风险等级、行为类别。
2. 明确区分 `normal`、`anomalous`、`not_applicable`、`skipped`、`error`、`timeout`。
3. 建立数据库 migration 目录和执行方式。
4. 准备完整 MySQL schema，不再依赖 repository 初始化时自动 `ALTER TABLE`。
5. 新增 scan batch、scan record、behavior event、evidence 等 repository 雏形。
6. 让 pipeline 能生成批次记录和每阶段检测记录。
7. 保存失败、超时、跳过、不可适用的检测记录，不要只保存 alive proxies。
8. 修复 API 和前端中的中文乱码，内部状态值使用稳定英文枚举，前端负责中文展示。
9. 前端拆出 API client、types、基础页面结构，避免继续把复杂逻辑堆在单个 `App.tsx`。
10. 为现有基础检测流程补最小回归测试。

第 0 阶段的目标不是完成所有功能，而是防止后续功能建立在不稳定的语义和表结构上。

## 数据库实施规则

用户已经确认：

1. 允许使用 MySQL 外键。
2. 希望一次性创建完整规划表结构，避免后续频繁改表导致数据丢失风险。
3. 当前项目已有 MySQL 连接配置。
4. 已测试到的数据库为 `proxy_pool`，MySQL 版本为 `8.0.45`。

因此请遵守：

1. 先写 SQL migration 文件，不要直接在业务代码中隐式改表。
2. 建表、重建表、删表、迁移已有数据都属于写操作，执行前必须得到用户明确授权。
3. 如果要重建或大幅调整 `proxies` 表，必须先备份已有数据。
4. `proxies` 主表只保存最新汇总状态，不保存完整检测过程。
5. 检测过程保存到批次表、记录表、事件表、证据表、证书观测表、资源观测表等明细结构。
6. 使用外键时要谨慎选择删除策略，避免误删研究历史。

建议迁移文件：

```text
migrations/
  001_extend_proxies_security_fields.sql
  002_create_security_scan_batches.sql
  003_create_security_scan_records.sql
  004_create_security_behavior_events.sql
  005_create_security_evidence_files.sql
  006_create_security_certificate_observations.sql
  007_create_security_resource_observations.sql
  008_create_proxy_sources.sql
  009_create_proxy_check_records.sql
  010_create_honeypot_targets.sql
  011_create_honeypot_request_logs.sql
```

完整 schema 可以一次性建全，但业务能力应分阶段启用。

## 安全检测实施规则

安全检测必须采用漏斗式逐层检测，不要默认对所有代理执行最重的检测。

推荐路径：

```text
基础连通性
  -> 协议识别
  -> 轻量蜜罐检测
  -> 内容 hash 对比
  -> DOM / HTML 风险规则
  -> 资源完整性检测
  -> HTTPS / SOCKS MITM
  -> 多轮条件触发检测
  -> 无头浏览器深度检测
  -> 行为分类与安全评分
```

必须注意：

1. 纯 HTTP 代理不应被标记为“MITM 正常”，应记录为 `not_applicable`。
2. 资源请求失败不应直接判定为资源替换。
3. 网络失败、超时、内容异常、证书异常必须区分。
4. 高成本检测只应对高可用、可疑、高价值或抽样代理执行。
5. “未检测”永远不能被当成“安全”。

2. 拆分目录结构。
3. 建立路由和 AppShell。
4. 拆分 API client。
5. 拆分类型定义。
6. 重做代理列表页。
7. 新增安全总览页。
8. 新增代理详情页。
9. 实现 `FunnelStepper`。
10. 增加折线图、柱状图、漏斗图、分布图等基础图表。
11. 新增单独的世界地图页面 `/map`。

世界地图第一版要求：

1. 国家级聚合即可，不必一开始做城市级。
2. 支持 hover tooltip。
3. tooltip 显示代理总数、活跃数、未检测数、正常/可疑/恶意数量、平均响应时间、协议分布、最高风险等级、主要异常类型。
4. 点击国家后打开区域详情面板。
5. 支持跳转到筛选后的代理列表或事件列表。

图表和地图建议优先考虑 ECharts，因为它可以同时覆盖折线图、柱状图、漏斗图和世界地图，集成成本相对较低。

## 文档优先级与去重执行规则

四份文档之间会有少量交叉提醒，这是为了保证核心原则一致，不代表同一任务需要重复执行。请按下面规则处理：

1. `docs/project-architecture-audit.md` 只用于识别当前结构问题和第 0 阶段前置整改项，不要把其中每条建议都扩展成完整功能实现。
2. `docs/next-build-plan.md` 只用于确定总体能力边界、检测顺序和阶段路线。
3. `docs/database-design-plan.md` 只在数据库、migration、repository、持久化模型相关任务中作为 schema 依据。
4. `docs/frontend-redesign-plan.md` 只在前端页面、组件、图表、地图、交互和展示语义相关任务中作为实现依据。
5. 如果多个文档都提到同一件事，只执行一次，并以本文档中的推荐实施顺序为准。
6. 如果一个建议既出现在审计文档，也出现在功能规划文档，应先把它理解为前置约束，再判断本轮任务是否真的需要实现。
7. 如果一个建议既涉及数据库又涉及后端或前端，应先完成数据模型和 API 契约，再实现调用方。
8. 数据库写操作、重建表、迁移已有数据、安装新依赖、启动长期任务等高影响操作，即使多个文档都建议执行，也必须先获得用户明确授权。
9. 每一轮实现前请先声明本轮只做哪些事项、明确不做哪些事项，避免把规划文档中的后续阶段内容提前混入当前任务。
10. 文档之间若出现冲突，以 `docs/next-thread-build-prompt.md` 的执行规则为最高优先级；仍然不确定时，先向用户说明冲突并请求确认。

## 验收标准

第一版安全检测 MVP 的验收标准：

1. 能对同一个蜜罐 URL 做直连和代理访问。
2. 能保存访问摘要。
3. 能对 HTML 和资源做 hash 对比。
4. 能识别明显 DOM 注入。
5. 能记录检测适用性、执行状态和跳过原因。
6. 能把安全风险汇总到代理。
7. 能保存检测批次、检测记录和异常事件。
8. 前端能展示安全风险、行为分类、漏斗路径。
9. 前端能用至少两个图表展示安全态势。
10. 前端有世界地图基础页面，支持国家级 hover 摘要。

## 最后提醒

这不再是一个单纯“代理可用性检测工具”。请始终围绕下面这句话做判断：

> ProxyTester 的下一阶段目标，是把代理从“可用/不可用”的资源对象，升级为“正常/可疑/恶意”的行为研究对象。

每一次代码修改都应该服务于这个目标。
